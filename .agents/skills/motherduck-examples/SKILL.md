# MotherDuck Examples

Use this skill before changing the MotherDuck examples catalog, README metadata,
Flight Plans, catalog generation, catalog validation, or starter download flow.

## Repository Model

This repo is becoming a flat, agent-readable catalog of MotherDuck examples.
Each catalog entry is a folder with working code and a README that serves both
humans and agents.

There are two entry families:

- `flight-plans/`: Flight-deployable plans. These use `features: [flights]` and
  deploy as MotherDuck Flights.
- Top-level example folders: normal examples such as dbt projects, app demos,
  ingestion scripts, UI walkthroughs, and integrations.

Keep non-flight examples at the repo root. Do not create an `examples/` wrapper
folder.

## Durable Decisions

- README YAML front matter is the catalog metadata source of truth.
- Shared dbt Flight logic for the current catalog lives in
  `flight-plans/dbt-runner`.
- Flight Plans are non-deterministic plans an agent adapts after asking the user
  a few questions.
- Flight deployment guidance belongs in README prose and MotherDuck MCP Flight
  tool usage, not checked-in create-flight SQL.
- `catalog.json` is generated by CI and served from GitHub Pages. It is not
  committed as a generated file.
- The catalog schema is committed in `catalog.schema.json`.

Vercel-style front matter and skill docs are useful inspiration, but the local
direction is README front matter plus a generated Pages catalog, not Contentful
sync or committed generated JSON.

## README Metadata

Catalog READMEs start with YAML front matter. For catalog entries, keep exactly
these six keys:

```yaml
---
title: Build Hacker News Models From S3 With dbt
id: dbt-ingestion-s3
description: >-
  Query public Hacker News Parquet data in S3 and build dbt models. Use when a
  user wants to ingest S3/Parquet with dbt or deploy a dbt build as a Flight.
type: example
features: [flights]
tags: [dbt, s3, parquet, hacker-news]
---
```

Rules:

- `id` must match the folder name.
- `type` is `example` or `template`.
- `features` is a subset of `dives`, `ducklake`, `flights`, `pg_endpoint`,
  `wasm`.
- `tags` are lowercase slugs for tools, datasets, and topics.
- Do not use `motherduck` or `dbt-duckdb` as tags.
- READMEs under `flight-plans/` must include `flights`.
- READMEs outside `flight-plans/` must not include `flights`.
- Nested or supporting READMEs may use `catalog: false` to opt out before normal
  required-field validation.

## README Body Pattern

Write every catalog README like a compact agent skill. Keep it practical and
derived from the actual files in that folder.

Use this structure:

1. `# <title>` and a short paragraph explaining what the example does and which
   MotherDuck pattern it demonstrates.
2. `## What you'll adjust`: a table of real knobs found in the code, with each
   knob's purpose and example values/options.
3. `## Questions to ask the user`: the information an agent should gather before
   adapting the example.
4. `## Run it`: local commands and environment variables.
5. For Flight Plans only, include `### Deploy as a Flight`.
6. `## How it works / Learn more`: link to extra files in the folder and point
   to MCP guides such as `get_flight_guide`, `get_dive_guide`, or
   `ask_docs_question`.

Do not put every possible explanation in the README. Prefer progressive
disclosure through nearby files and MCP guide pointers.

## Catalog Tooling

The catalog builder is [scripts/build-catalog.py](../../../scripts/build-catalog.py).
It validates front matter by default and writes JSON only when `--output` is
provided.

Use:

```bash
uv run scripts/build-catalog.py
uv run scripts/build-catalog.py --output .catalog-pages/catalog.json
uv run --with check-jsonschema check-jsonschema --schemafile catalog.schema.json .catalog-pages/catalog.json
```

Run tests with:

```bash
uv run --with pytest --with pyyaml --with jsonschema python -m pytest tests/test_build_catalog.py -q
```

The GitHub Actions workflow validates on PRs, uploads a `catalog-preview`
artifact, and deploys `catalog.json` plus `catalog.schema.json` to GitHub Pages
on pushes to `main`.

## Starter Script

[scripts/get-starter.sh](../../../scripts/get-starter.sh) must keep short-name
resolution for both top-level examples and `flight-plans/<id>`.

When testing a PR branch:

```bash
BRANCH=my-branch curl -fsSL https://raw.githubusercontent.com/motherduckdb/motherduck-examples/my-branch/scripts/get-starter.sh | bash -s dbt-ingestion-s3
```

## Working Guidance

- Before rewriting a README, read the example's code, config, and current README
  enough to identify real adjustable parts.
- Keep README front matter minimal. Move deployment details, schedules,
  requirements, parameters, and caveats into prose.
- Use `uv` for Python tooling.
- Use `gh` for GitHub work when needed.
- Prefer DuckDB SQL examples and MotherDuck terminology that match the existing
  docs style.
- Preserve unrelated user or agent changes in this Conductor workspace.
